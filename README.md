# TranslateBook

AI-assisted modular literary translation pipeline for TXT documents.

## Literary Translation Intelligence Phase

This phase upgrades the MVP with literary quality controls while preserving the original modular CLI architecture.

### New capabilities

- Strict style profile system (`style_profile.json`)
- Book-level context memory across chunks (per run)
- Glossary policies (`preferred` and `forced`) with relevance matching
- Dialogue-aware / narration-aware / emphasis-aware prompt behavior
- Optional 2-stage translation flow:
  - Stage 1: faithful draft
  - Stage 2: literary refinement (OpenAI only, mock is no-op)
- Consistency-focused sidecar report generated by default (`<output>.report.json`)
- Cache/TM partitioning by policy hash (style + glossary + refinement + context window)

## Scope

- Input: TXT only
- Output: TXT only + JSON report
- No EPUB/PDF/UI in this phase

## Requirements

## CLI Usage

### Baseline (backward compatible)

```bash
python main.py --input data/input/book.txt --output data/output/book_es.txt
```

### Literary profile + glossary + refinement

```bash
python main.py \
  --input data/input/sample_literary_excerpt.txt \
  --output data/output/sample_literary_es.txt \
  --translator openai \
  --api-key sk-... \
  --style-profile data/style_profiles/example_literary_es.json \
  --glossary data/glossaries/example_literary_es.json \
  --enable-refinement \
  --context-window 3
```

### Custom report location

```bash
python main.py \
  --input data/input/book.txt \
  --output data/output/book_es.txt \
  --report-output data/output/book_es.report.json
```

### Disable reuse layers

```bash
python main.py --input data/input/book.txt --output data/output/book_es.txt --disable-cache --disable-tm
```

## CLI Arguments

### Required

- `--input`, `-i`: input TXT path
- `--output`, `-o`: output TXT path

### Core optional

- `--chunk-size`: target words per chunk (100-5000, default `1000`)
- `--translator`: `mock` or `openai` (default `mock`)
- `--api-key`: OpenAI API key (or use `OPENAI_API_KEY`)
- `--model`: OpenAI model (default `OPENAI_MODEL` or `gpt-5.2`)
- `--source-language`: source language code (default `en`)
- `--target-language`: target language code (default `es`)
- `--verbose`, `-v`: verbose logs

### Literary phase optional

- `--style-profile`: style profile JSON path
- `--glossary`: glossary JSON path
- `--enable-refinement`: enable stage-2 refinement
- `--context-window`: number of recent chunk memories injected into prompts (default `3`)
- `--cache-db`: SQLite path for cache + TM
- `--disable-cache`: disable cache layer
- `--disable-tm`: disable translation memory layer
- `--report-output`: report output JSON path (default `<output>.report.json`)
- `--report`: deprecated alias for `--report-output`

## Style Profile

A strict JSON object with required fields:

- `source_language`
- `target_language`
- `genre`
- `tone`
- `audience`
- `formality`
- `translation_strategy`
- `author_voice_notes`
- `dialogue_style`
- `narrator_style`
- `forbidden_patterns` (non-empty list)
- `preferred_patterns` (non-empty list)

Sample: `data/style_profiles/example_literary_es.json`

## Glossary Format

Backward-compatible format with optional new controls:

```json
{
  "terms": [
    {
      "source": "Princess Donut",
      "target": "Princesa Donut",
      "notes": "Keep name identity",
      "policy": "forced",
      "case_sensitive": false
    }
  ]
}
```

- `policy`: `preferred` or `forced` (default `preferred`)
- `case_sensitive`: source/target matching sensitivity (default `false`)

## Pipeline Flow

Per chunk:

1. Analyze chunk signal (`narration`, `dialogue`, `emphasis`, `mixed`)
2. Detect relevant glossary terms
3. Build prompt with style profile + context memory + chunk signal + glossary rules
4. Try cache
5. Try translation memory
6. Stage 1 translation
7. Optional Stage 2 refinement
8. Run consistency heuristics and warnings
9. Update context memory
10. Persist final translation to cache/TM

## Report Output

Default path: `<output>.report.json`

The report includes:

- run metadata (files, languages, model, timings)
- chunk counts and reuse stats
- style profile metadata and hashes
- glossary relevant/used counters
- recurring discovered terms
- consistency risks and uncertain translation warnings
- per-chunk diagnostics (signal type, stage info, glossary use, warnings)

## Quality Gate

Run a deterministic quality gate using the generated JSON report:

```bash
python main.py \
  --input data/input/sample_literary_excerpt.txt \
  --output data/output/ci_sample_es.txt \
  --translator mock \
  --style-profile data/style_profiles/example_literary_es.json \
  --glossary data/glossaries/example_literary_es.json \
  --enable-refinement \
  --cache-db data/cache/ci_translation_store.sqlite3 \
  --report-output data/output/ci_sample_es.report.json

python quality_gate.py \
  --report data/output/ci_sample_es.report.json \
  --max-errors 0 \
  --max-risks 0 \
  --max-uncertain 0 \
  --max-warning-chunks 0 \
  --max-refinement-drift 0.55
```

CI automation is included in `.github/workflows/ci.yml`.
